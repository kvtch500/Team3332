# Live Activity (Android analog) setup — one-time wiring

This sets up the **Android counterpart** of the iOS run Live Activity (618g/619): an **ongoing
notification** that shows live **distance / time / pace** while a run records, with a self-ticking
timer, a "Run complete" end-flash, and a tap-to-open deep link. The Kotlin plugin and the JS bridge
are already written — the JS side (`app/src/app.jsx`'s `LiveActivity` helper) is **shared with iOS**
and needs **no changes**. This doc is the native wiring you do once on the Mac.

> Prerequisite: the Android project must exist. If you haven't generated it yet, do
> `mobile/SETUP-ANDROID.md` Steps 0–2 first (`npx cap add android` + the location permissions). This
> doc adds the notification feature on top.

> **Where the source lives:** `mobile/android/` is **gitignored** — it's regenerated by
> `npx cap add android`, so files placed inside it are NOT committed and can be wiped. The durable,
> committed source-of-truth is **`mobile/android-native-src/`**. If you ever re-run `cap add android`,
> re-do Steps 1–3 below.

## Step 1 — Add the plugin + register it

1. Copy the plugin into the project (paths match 1:1):

   ```bash
   cd "/Users/ernestsmith/Desktop/claude ai/3332/mobile"
   cp android-native-src/app/src/main/java/com/team3332/app/LiveActivityPlugin.kt \
      android/app/src/main/java/com/team3332/app/LiveActivityPlugin.kt
   ```

2. Register the local plugin in `MainActivity`. Either copy the committed version over the generated
   one, or add the single `registerPlugin(...)` line by hand:

   ```bash
   cp android-native-src/app/src/main/java/com/team3332/app/MainActivity.java \
      android/app/src/main/java/com/team3332/app/MainActivity.java
   ```

   The key line (before `super.onCreate`):
   ```java
   registerPlugin(LiveActivityPlugin.class);
   ```

   > Android has **no `packageClassList`** — this `registerPlugin` call is the entire registration
   > mechanism (the analog of the iOS Step 5 gotcha). Miss it and `registerPlugin('LiveActivity')`
   > in JS resolves to nothing: the notification silently never appears, no error, no log.

## Step 2 — Manifest entries

Open `android/app/src/main/AndroidManifest.xml` and apply
`android-native-src/AndroidManifest-additions.xml`:

1. **`POST_NOTIFICATIONS`** (Android 13+) — required to show any notification. It's already in the
   SETUP-ANDROID Step 2 list for the background-GPS notification; if you followed that, it's there.
   The run-stats notification rides the **same** grant — no extra runtime prompt.
2. **`team3332://run` deep link** — add the `<intent-filter>` from the additions file *inside* the
   existing `<activity android:name=".MainActivity" …>` element, next to the `LAUNCHER`
   intent-filter. This lets tapping the notification open the app to the record screen via the
   `@capacitor/app` `appUrlOpen` handler already in `app.jsx` (same URL as the iOS card).

## Step 3 — (Optional) custom notification icon

The plugin uses the built-in `android.R.drawable.ic_menu_compass` as a safe default small icon. For
a polished status-bar glyph, add a **white, transparent, 24dp** `ic_stat_run` to
`android/app/src/main/res/drawable*/`, then change `smallIcon()` in `LiveActivityPlugin.kt` to
return `R.drawable.ic_stat_run`. Not required to ship.

## Step 4 — Build & run


## How to test on device / emulator

1. Grant **notifications** (and location "Allow all the time") when prompted.
2. **Record** a run → **START**.
3. Pull down the shade / lock the screen: you should see the **TEAM 3332 — Run** notification with
   **`<distance> mi · <pace> /MI`** and a **ticking timer** (top-right). The time keeps moving on its
   own; distance/pace refresh as the recorder pushes updates.
4. **Tap** the notification → the app opens to the **record** screen (deep link).
5. **END** the run → the notification flips to **"✓ Run complete"** with frozen final stats for ~4s,
   then clears. Leaving mid-run (without ending) clears it immediately.

## Troubleshooting

- **No notification at all** → (a) plugin not registered — check the `registerPlugin` line in
  `MainActivity` (Step 1.2); (b) `POST_NOTIFICATIONS` denied — check **Settings → Apps → TEAM 3332 →
  Notifications**.
- **Notification shows but time doesn't tick** → confirm `setUsesChronometer(true)` and a sane
  `setWhen(baseWhenMs)` (start anchors it; don't reset it on every update).
- **Tap doesn't open the record screen** → the `team3332` `<intent-filter>` is missing or outside the
  `MainActivity` element (Step 2.2). The notification still opens the app; only the auto-jump to the
  record screen depends on the filter + the JS `appUrlOpen` handler.
- **Duplicate-looking notifications** → the plugin reuses a fixed id (`NOTIF_ID = 33321`); every
  `update()` replaces the same notification. If you see two, another code path is posting to a
  different id.

## Notes

- **Shared JS, zero changes.** `app/src/app.jsx` calls `LiveActivity.start/update/end` on any native
  platform; on Android those reach this plugin, on iOS the Swift one. Rejections are swallowed by the
  JS `safe()` wrapper, so a notification hiccup can never interrupt recording.
- **Payload contract** (set by `liveActivityPayload()` in `app.jsx`): `activityType` ("Run"|"Walk"),
  `distanceMiles` (Double), `elapsedSeconds` (Int), `metricValue` (String, e.g. "8:42"),
  `metricLabel` (String, "/MI" or "MPH"). The Kotlin plugin reads exactly these.
- **Not a foreground service** — background-geolocation already owns the tracking service and keeps
  the process alive; this is just the live-stats notification.
- The sandbox can't compile Kotlin, so this was validated by static/structural review only — first
  real compile happens here in Android Studio.
